<h2 id="why-ctms-core">Why ctms-core</h2>

<div class="landing-grid">
  <div class="landing-card">
    <h3>A real relational core</h3>
    <p>
      Documents are rows, typed by the CDISC TMF Reference Model taxonomy and
      scoped to a study, a study-site, or a person, with immutable
      content-addressed versions. Studies, sites, people, and role assignments
      are first-class entities, not folder names.
    </p>
  </div>
  <div class="landing-card">
    <h3>A requirement engine</h3>
    <p>
      Declarative rules materialize <em>expected</em> documents: every active
      site needs a current IRB approval, every investigator a CV under 2 years
      old. Status (missing, pending review, current, expiring soon) is derived
      by views on every read, so it cannot drift from the documents themselves.
    </p>
  </div>
  <div class="landing-card">
    <h3>Compliance as schema, not as feature</h3>
    <p>
      An append-only, hash-chained audit trail written by triggers on every
      mutation; document versions that cannot be updated or deleted; signatures
      cryptographically bound to the content hash they signed. An auditor's
      question is also just a query.
    </p>
  </div>
  <div class="landing-card">
    <h3>The API is the product</h3>
    <p>
      The dashboard consumes only the public REST API, so anything the UI shows,
      your script can query. The derived-status views (<code>v_*</code>) are
      documented public surface too: R or Python reads the same computed truth
      over a read-only Postgres connection. See the
      <a href="/ctms-core/cookbook/">cookbook</a>.
    </p>
  </div>
  <div class="landing-card">
    <h3>Operations, the same way</h3>
    <p>
      Monitoring visits, protocol deviations, enrollment, and milestones are
      dated facts in tables, with lifecycle stages (overdue, awaiting report,
      resolved) derived by views. No status column anywhere in the system. See
      the <a href="/ctms-core/operations/">operational layer</a>.
    </p>
  </div>
  <div class="landing-card">
    <h3>Open, with a held boundary</h3>
    <p>
      AGPL-3.0, with CDISC eTMF-EMS exchange in both directions. Subject-level
      clinical data stays in your EDC, which files into the TMF through the
      documented upload API; the reference client is
      <a href="https://github.com/tgerke/edc-core">edc-core</a>, this project's
      sibling on the data-capture side.
    </p>
  </div>
</div>
